## 概要

モックは 3 画面しかない。残り 7 画面は 19 で確立した型を借りて揃える。**新しい部品を発明しない**のが本チケットの唯一のルール。

## 仕様

### 全ページ共通

- セクション見出しは `SectionHeading`（§5.0）に統一
- 面構成（§4.2）を適用。**生成り一色で最後まで行かない**
- 一覧ページの Hero は写真が無いので `display-xl` 96px + kicker + 英字サブ（カレンダーページと同じ型）

### 個別

- **工芸一覧**: カードに `No. 01` の通し番号（一覧内の並び順ではなく**工芸ごとに固定の番号**を振る。並べ替えで番号が変わらないこと）
- **体験一覧**: 既存の `ExperienceFilterList`（`'use client'` + URL 同期）を壊さない。フィルタチップに件数バッジを追加
- **記事一覧**: 既存の `ArticleFilterList` を同様に維持 + 件数バッジ
- **イベント詳細**: 申込カードを `sticky` に。外部申込リンクは「申し込む（外部サイト）」と**遷移先が外部であることを明示**（DESIGN §5.1）。終了イベントはアーカイブ表示を維持
- **記事詳細**: `.article-content` のタイポは維持。`hr` の糸＋結び目もそのまま。**ページ枠だけ v0.2 に合わせる**（ヘッダー / フッター / 面構成）
- **About**: フッターから `/about#operator` `/about#contact` のアンカーが張られる（§5.13）。**この 2 つの見出しに `id` を必ず用意する**
- **Privacy**: 装飾は最小限。読みやすさ優先

### 横断の確認

- `EventRow` / `ExperienceCard` / `ArticleCard` の任意 prop `craft?`（一覧でのみ所属工芸を表示）が v0.2 でも機能すること
- アンカー入れ子回避のパターン（タイトル Link の `after:absolute after:inset-0` + 内側 Link の `z-10`）を維持すること
- `is_provisional` の「※確認中」表示が全ページで維持されること

### 実装で決めたこと

- **通し番号の正本を関数に固定** — `src/lib/craftNumber.ts` の `craftNumberMap(crafts)`（`getCrafts` の `created_at` 昇順が正本）。ホーム・工芸一覧・工芸詳細の 3 箇所がこれを使う。**絞り込み後のリストを渡さない**ことをコメントで明示した
- **スタッツは「帯」で置いた**（カード化しない） — 当初カレンダーと同じ `PageHero` の `aside` にカード化したが、**h1 の欄が 626px に痩せて `display-xl` が 5 文字で折り返し**、`.mark-gold` が行をまたいで割れた。DESIGN §6 の文言も「display-xl + スタッツ**帯**」なので、ホーム #2 と同じ `border-t` + `Stat` の全幅帯に変更。カード化する型はカレンダーページ（Hero 右に置ける）だけの扱いにする
- **和文見出しの改行は文言側**（19 の決定を踏襲）。`Crafts.listHeading` 等に `<br></br>` を入れ、ページ側は `t.rich` の `br` ハンドラで解決。EN は空白で自然に折り返るので `<br>` を入れない
- **ピル型ボタンのラベルに可変長のデータを入れない** — 記事詳細の締めの帯で工芸名をボタンラベルにしたところ、EN の工芸名が長く `whitespace-nowrap` のため 375px で**ページ全体が横スクロール**した（実測 401px > 375px）。固定文言（`outroCtaCraft`）に変更し、工芸名は本文側に出す
- **Privacy と 404 の h1 は `display`(74px) ではなく `h1`(44px)** — 読み物幅 720px に 74px を置くと「プライバシーポリ／シー」のようにカタカナ語の途中で折り返る。§6 の「Privacy は装飾最小限・読みやすさ優先」を優先した
- **Privacy は面 1 のみ**（§4.2 の「最低 3 種類」より §6 の個別指定を優先）。条項見出しにも `SectionHeading`（40px の 3 層見出し）は使わない。6 連続すると読み物として破綻するため。**「セクション見出しは `SectionHeading` に統一」はセクション見出しの話で、ポリシーの条項見出しは対象外**という解釈
- **絞り込みチップを共通化** — `EventsView` にあった件数バッジ付きチップを `ui/FilterChip` に切り出し、体験一覧・記事一覧・カレンダーの 3 箇所で共有（見た目は不変）。件数は「**他の群の絞り込みを適用した状態での該当件数**」＝押した結果の件数
- **スタッツの集計を `lib/data/stats.ts` に切り出し** — 一覧 3 画面が同じ数字を必要とするため。`getHomeData` もこれを使う（`HomeData.stats` の型・ホームの表示は不変）。個々の取得関数が `cache()` 済みなので追加クエリは発生しない
- **EN フォールバックバナーは現行シードでは発火しない** — `seed.sql` の `*_translations` は **EN も全件 `is_published = true`** なので `isFallback` が立たない。条件（`locale === 'en' && x.isFallback`）は 3 つの詳細ページすべてに残してあり、docs/15 で EN 未訳のコンテンツが入った時点で出る

### 検証結果

- `npm run lint` / `npm run build` が通り、**公開 10 画面すべてが ● SSG のまま**（`/[locale]/[...rest]` だけが ƒ Dynamic。`notFound()` を呼ぶだけのキャッチオールで従来どおり）
- **横スクロール実測（CDP）**: 今回の 8 URL（一覧 3 / 詳細 2 / About / Privacy / 404）× `ja`・`en` × 375・768・1440px = **48 通りで横スクロールなし**。19 の 3 画面（ホーム / 工芸詳細 / カレンダー）も含めた 66 通りでも 0 件（共通部品の差し替えによる回帰なし）
- **ハイドレーション後の挙動**: 体験一覧のチップが件数バッジ付きで動き `?craft=` に同期、記事一覧が `?craft=ajima-gasa` から初期化、カレンダーのチップと表示切替（`?view=calendar`）も従来どおり
- `prefers-reduced-motion: reduce` で transition が 0.15s → 1e-06s になり、`Reveal` が要素を隠さない（`opacity:0` の要素数 2 → 0）
- `/about#operator` `/about#contact` はスティッキーヘッダーの下（`scroll-mt-28` = 112px）に着地する
- 写真ゼロでもレイアウトは破綻しない（工芸カード・記事カードは `CardMedia` のプレースホルダ、イベント詳細は写真枠そのものを置かない）

### 実装後の構造監査（CDP で DOM を検査）で直したもの

見た目の確認だけでは出てこない構造の不具合を、全 11 画面に対して機械的に検査した（見出し階層 / `dl` の内容モデル / 重複 id / 名前のないリンク / `alt` 欠落）。

| 見つかったもの | 影響 | 対応 |
|---|---|---|
| **`Stat` の `dl` が `dd` → `dt` の順**（HTML の内容モデル違反。docs/17 由来） | 支援技術が用語と説明を取り違える可能性。露出はスタッツ帯のある 5 画面 | DOM を `dt` → `dd` に直し、`flex-col-reverse` で**見た目だけ**従来どおり（数字が上）に保った |
| **一覧 3 画面で見出しが h1 → h3 に飛ぶ** | カードの見出しが h3 なのに、あいだの h2 が無く**アウトラインが崩れる**（Hero の h1 直下にカード） | グリッドの前に `sr-only` の h2 を置いた。視覚的には Hero と重複するので読み上げ・アウトライン専用 |
| **工芸詳細に h1 が 2 個**（docs/19 の PC / SP 出し分けで DOM が重複） | 表示・読み上げは常に片方だけだが、DOM 上は 2 つ | `CraftHero` のタイトルブロックを **1 つに統合**し、位置（`static` ⇄ `lg:absolute`）と配色だけレスポンシブに切り替える形へ。375 / 768 / 1440px でスクリーンショットが**リファクタ前と同一**であることを確認済み |
| **`getArticles(locale, {})` の `{}` で `cache()` が効いていなかった** | `cache()` は引数の同一性でキーを引くため `{}` は毎回別キー。ホームと `getSiteStats` で同じクエリが 2 回走っていた | 絞り込みの要らない呼び出しは第 2 引数を渡さない形に統一（`stats.ts` / `home.ts`）。`SiteStats` から未使用の `upcomingEvents` も削除し、一覧 3 画面のクエリを 1 本減らした |
| 未使用の文言キー（`Experiences.price` / `Home.upcomingTitle`） | — | 削除（ボーイスカウトルール） |
| **記事詳細の本文が h1 → h3 に飛ぶ** | `.article-content` に流し込むシードの HTML が `<h3>` から始まっていた（コードではなく**コンテンツ側**） | `seed.sql` の記事本文を `<h2>` 始まりに修正し、リモート DB にも `update ... replace(content,'<h3>','<h2>')` で反映（4 行）。**docs/15 で本文を書くときも h2 から始める**（`.article-content` は h2/h3/h4 のスタイルを持つ） |
| **イベント詳細で会場・料金が 3 箇所に出ていた**（Hero のメタ行 / 本文の dl / 申込カード） | 同じ事実が 3 回並び、読み手が突き合わせる羽目になる | **Hero のメタ行を削除**し、「本文の dl = 全項目 / 申込カード = 申込サマリ」の 2 役に整理。日付だけは kicker（`Event · 8月23日(土) · 遠山ふじ糸`）に入れて Hero でも分かるようにした |

> **DB のコンテンツを更新したら `rm -rf .next` のクリーンビルドが要る**（docs/06 のメモと同じ現象）。通常のビルドでは記事本文が更新前のまま出力された。

**残っているコンテンツ側の既知事項**: シードの `description` 自体に「※確認中」が含まれており、`is_provisional` の表示と合わせて 2 回出る画面がある。docs/15 の本番コンテンツ差し替えで解消する。
